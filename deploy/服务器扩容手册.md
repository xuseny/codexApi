# Sub2API 服务器扩容手册

本文档最初基于 2026-04-02 的双机扩容结果整理，并已在 2026-04-10 主节点迁移完成后按现网拓扑校正，目标是把当前二进制部署方式沉淀成一份可复用手册，供后续继续横向扩容时直接参考。

适用前提：

- 当前部署方式为 `systemd + Caddy + PostgreSQL + Redis + sub2api`
- `sub2api` 以二进制方式运行
- 数据库和 Redis 仍然集中部署在主节点
- 新增节点只承担应用层流量，不单独部署 PostgreSQL/Redis/Caddy

说明：

- 本文最初沿用 `A / B / C` 命名记录双机扩容过程
- 截至 2026-04-10，`xuseny.online` 的实际主节点已经迁移到新服务器 `C（154.64.230.47）`
- 旧 `A（156.239.40.153）` 现在只应视为迁移过渡期的临时 HTTPS 代理 / 回滚入口，不再视为长期主节点
- 阅读本文中后续的 `A 机` 表述时，应优先理解成“当前主节点角色”

## 1. 当前现网结构

截至 2026-04-10，当前线上结构如下：

```text
域名:
  xuseny.online
  www.xuseny.online
  api.xuseny.online

C 机（当前主节点，154.64.230.47）
  - Caddy
  - sub2api
  - PostgreSQL
  - Redis

B 机（应用节点，10.0.69.3）
  - sub2api

旧 A 机（156.239.40.153）
  - 仅在 DNS / HTTPS 证书传播未完全稳定时保留
  - 只承担临时反代 / 回滚入口
  - 不再长期承担 PostgreSQL / Redis / 主应用职责
```

当前应用端口约定：

- C 机：`127.0.0.1:2127`
- B 机：`0.0.0.0:2127`

当前稳态下，Caddy 统一部署在当前主节点 C，由 C 对外提供 80/443，并把请求转发给：

- `127.0.0.1:2127`
- `10.0.69.3:2127`

## 2. 本次扩容的最终状态

在“先双机扩容，再完成主节点迁移”之后，已经确认以下状态成立：

- C、B 两台 `sub2api` 服务都可以承担业务流量
- 域名健康检查成功：
  - `https://xuseny.online/health`
  - `https://api.xuseny.online/health`
- B 机已经收到真实业务流量，而不仅仅是健康检查
- 当前主节点上的 Redis 已经能够看到来自应用节点的连接

本次扩容中最终统一的关键配置如下：

- `jwt.secret`
- `redis.password`
- `totp.encryption_key`

当前仍建议保留在主节点处理的路由：

- `/api/v1/admin/data-management/*`

这样做的原因是：

- 数据管理相关能力仍然更适合固定在主节点
- 这样改动最小，回滚也最简单

## 3. 本次扩容踩过的关键问题

后续继续扩容时，优先避开下面这些问题：

### 3.1 不要以公网/NAT 地址作为服务监听依据

服务器真正绑定和互通时，应优先使用机器实际内网地址，例如：

- A：`10.0.69.2`
- B：`10.0.69.3`

不要把公网映射地址当作本机监听地址写入服务配置。

### 3.2 应用节点不要重复部署数据库和 Redis

B 机是纯应用节点，因此：

- 不需要单独安装 PostgreSQL
- 不需要单独安装 Redis
- 不需要单独安装 Caddy

否则排障时很容易把“主节点命令”误执行到应用节点。

### 3.3 Redis 需要明确支持跨节点访问

本次问题的核心之一是：

- B 可以连到 A 的 Redis 端口
- 但 Redis 因 `protected mode` 拒绝远程访问

最终修复方式是：

- 保持 Redis 只部署在 A
- 为 Redis 设置密码
- 让 B 使用统一密码访问 A 的 Redis

### 3.4 共享密钥必须统一

多节点部署时，下列值必须一致：

- `jwt.secret`
- `redis.password`
- `totp.encryption_key`

如果这些值不一致，会导致：

- 登录态不一致
- 跨节点行为异常
- TOTP/2FA 数据不可互通

### 3.5 配置文件权限必须正确

本次 B 机曾出现：

```text
Failed to load config: open /opt/sub2api/config.yaml: permission denied
```

因此后续每次同步配置后，都应检查：

- `/opt/sub2api/config.yaml`
- `/opt/sub2api/sub2api`

推荐权限：

- 目录：`755`
- 二进制：`755`
- 配置文件：`640`
- 文件属主：`sub2api:sub2api`

### 3.6 不要用 `root` 前台运行 `caddy run` 代替 systemd

本次主节点迁移时踩到的一个非常关键的问题是：

- `systemctl start caddy` 使用的是 `caddy` 用户的存储目录
- `root` 手工执行 `caddy run --config /etc/caddy/Caddyfile` 使用的是 `root` 自己的 Home 目录

这会导致你在前台调试时看到的证书状态，与 systemd 实际运行时使用的证书状态不一致。

排查 Caddy 证书问题时，优先使用：

- `systemctl status caddy --no-pager -l`
- `journalctl -u caddy -n 100 --no-pager`
- `find /var/lib/caddy/.local/share/caddy/certificates -type f`

不要把 `root` 前台运行的 Caddy 结果，直接当成线上最终状态。

### 3.7 主节点迁移切 DNS 后，旧公网 IP 可能仍会收到 ACME 校验流量

本次主节点从旧 A 迁移到新 C 后，即使 DNS 已改到新 IP，Let's Encrypt 的校验流量和部分客户端流量在一段时间内仍然命中了旧公网 IP。

实际有效的过渡方案是：

- 新主节点 C 正常运行 `sub2api + Caddy`
- 旧 A 不再跑 `sub2api`
- 旧 A 仅保留 `Caddy`
- 旧 A 的 `Caddy` 临时把全部 HTTPS 请求反代到 `http://C_NEW_IP:80`

这样可以保证：

- 命中旧 IP 的用户请求仍能到达新主节点
- 命中旧 IP 的 ACME HTTP/TLS 校验也能被新主节点接住
- 等传播稳定后，再彻底下掉旧 A

## 4. 当前推荐的标准扩容原则

后续每次扩容，都按下面的原则执行：

说明：

- 下文继续沿用 `A = 当前主节点`、`B/C = 应用节点` 这种角色化写法
- 截至 2026-04-10，现网中这个“主节点 A 角色”实际对应到新服务器 `154.64.230.47`

### 4.1 主节点职责固定

A 机继续承担：

- 公网入口
- Caddy
- PostgreSQL
- Redis
- sub2api

### 4.2 新增节点只做应用节点

新加的机器只承担：

- `sub2api`

不额外部署：

- PostgreSQL
- Redis
- Caddy

### 4.3 生产配置以 A 机为准

所有新节点配置都从 A 机生产配置派生，只覆盖少量本机差异项：

- `server.host`
- `server.port`
- `database.host`
- `database.port`
- `redis.host`
- `redis.port`

### 4.4 新节点先验活，再加到流量池

每次扩容必须按顺序执行：

1. 启动新节点
2. 确认新节点 `/health` 正常
3. 确认它已连上主节点 Redis
4. 最后再改 Caddy

## 5. 下一次扩容方案：三服务器

下一次扩容建议采用“三服务器结构”：

```text
域名
  -> A 机 Caddy
      -> A 机 sub2api (127.0.0.1:2127)
      -> B 机 sub2api (10.0.69.3:2127)
      -> C 机 sub2api (10.0.69.4:2127)   # 示例地址

A 机（10.0.69.2）
  - Caddy
  - PostgreSQL
  - Redis
  - sub2api

B 机（10.0.69.3）
  - sub2api

C 机（建议 10.0.69.4）
  - sub2api
```

说明：

- `10.0.69.4` 是下一台应用机的建议示例地址
- 如果实际地址不同，按实际内网地址替换

## 6. 三服务器扩容的执行思路

### 6.1 C 机角色

C 机应与 B 机保持相同定位：

- 只跑 `sub2api`
- 不跑 PostgreSQL
- 不跑 Redis
- 不跑 Caddy

### 6.2 C 机需要同步的内容

从 A 机同步到 C 机的核心内容：

- `/opt/sub2api/sub2api`
- `/opt/sub2api/config.yaml`
- `sub2api.service`
- `override.conf` 或等效环境变量配置

### 6.3 C 机需要覆盖的配置项

C 机配置建议基于 A 的生产配置，仅覆盖以下内容：

```yaml
server:
  host: "0.0.0.0"
  port: 2127

database:
  host: "10.0.69.2"
  port: 5432

redis:
  host: "10.0.69.2"
  port: 6379
```

其余关键共享配置必须与 A/B 保持一致：

- `jwt.secret`
- `redis.password`
- `totp.encryption_key`

## 7. 三服务器扩容的推荐步骤

### 步骤 1：准备 C 机基础环境

保证 C 机至少具备：

- `sub2api` 运行用户
- `/opt/sub2api`
- systemd

### 步骤 2：从 A 机同步生产配置到 C 机

同步时以 A 机现网配置为准，不以仓库示例为准。

### 步骤 3：仅修改 C 机本机差异项

只调整：

- `server.host`
- `server.port`
- `database.host`
- `database.port`
- `redis.host`
- `redis.port`

### 步骤 4：修正权限并启动 C 机

启动前确认：

- 二进制属主正确
- 配置文件属主正确
- 配置文件可被 `sub2api` 用户读取

### 步骤 5：先做单机健康检查

C 机至少要通过以下检查：

- `systemctl status sub2api --no-pager -l`
- `ss -lntp | grep 2127`
- `curl http://127.0.0.1:2127/health`

### 步骤 6：确认 C 已连上 A 的 Redis

在 A 机确认 Redis 客户端列表里出现来自 C 的连接。

### 步骤 7：最后再把 C 加入 Caddy

确认 C 已稳定后，才把 C 节点加入 A 机 Caddy upstream。

## 8. 三服务器时的 Caddy 推荐写法

三机时，A 机 Caddy 中负载均衡段建议如下：

```caddy
xuseny.online, www.xuseny.online, api.xuseny.online {
    }

    @data_management {
        path /api/v1/admin/data-management/*
    }

        reverse_proxy 127.0.0.1:2127
    }

    handle @data_management {
        reverse_proxy 127.0.0.1:2127
    }

    handle {
        reverse_proxy 127.0.0.1:2127 10.0.69.3:2127 10.0.69.4:2127 {
            health_uri /health
            health_interval 30s
            health_timeout 10s
            health_status 200
            lb_policy round_robin
            lb_try_duration 5s
            lb_try_interval 250ms
        }
    }
}
```

说明：

- `10.0.69.3` 为 B
- `10.0.69.4` 为 C
- 如果实际 IP 不同，替换成真实内网 IP

## 9. 为什么暂时不建议把所有路由都放进轮询

当前仍不建议把以下路由直接放进三机轮询：

- `/api/v1/admin/data-management/*`

原因如下：

- 数据管理链路更适合固定在主节点
- 当前方案以“最小改动、最容易回滚”为优先

如果未来要彻底去掉这些固定路由，建议先完成下面任一改造：

- 把数据管理能力拆成独立可横向扩展的后端服务

## 10. 三服务器扩容后的验收标准

完成三机扩容后，建议至少验证：

### 10.1 服务状态

- A、B、C 三台 `sub2api` 全部运行中
- A 上 `Caddy` 运行中
- A 上 PostgreSQL、Redis 运行中

### 10.2 健康检查

- `http://127.0.0.1:2127/health`
- `http://10.0.69.3:2127/health`
- `http://10.0.69.4:2127/health`
- `https://xuseny.online/health`
- `https://api.xuseny.online/health`

### 10.3 运行时连通性

- A 的 Redis 中可以看到来自 B、C 的连接
- B、C 的应用日志中能看到来自 A 反代过来的真实请求

### 10.4 业务验证

至少验证以下能力：

- 登录态正常
- `/api/v1/settings/public` 正常
- `/v1/responses` 正常
- 真实业务请求可以命中不同节点

## 11. 回滚方案

如果三机扩容后出现异常，最简单回滚方式如下：

### 11.1 从 Caddy 中移除新节点

先把异常节点从 A 机 Caddy upstream 删除，只保留稳定节点。

### 11.2 重载 Caddy

让流量先回到剩余正常节点。

### 11.3 保留节点文件，暂不立即清理

不要第一时间删除新节点文件，优先保留：

- 二进制
- 配置
- 日志

这样更利于排查问题。

## 12. 后续建议

如果未来还会继续从 3 台扩到 4 台、5 台，建议保持同样策略：

- 数据层继续集中在 A
- 新增节点一律做 app-only
- 共享密钥始终从 A 同步
- 新节点先验活，再进流量池
- 需要本地文件的路由继续固定到 A，直到改成共享存储

如果后续规模继续扩大，再考虑下一阶段演进：

- PostgreSQL 主从或托管化
- Redis 高可用
- 数据管理能力独立化
- Caddy 前再引入更标准的 LB/网关层
